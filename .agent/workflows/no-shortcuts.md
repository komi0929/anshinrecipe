# 手抜き厳禁ルール

## 絶対原則

**「最小限」「とりあえず」「後で追加」は禁止。常に完全版を提供する。**

---

## 1. ダミーデータ・テストデータ作成時

### 必須チェックリスト

- [ ] **全カラムに値を入れる** - NULLが許容されるカラムでも、デモ目的なら値を入れる
- [ ] **配列・JSONBは複数要素を入れる** - 空配列や単一要素は禁止
- [ ] **画像URLは全て有効なものを使用** - Unsplashなど実際に表示される画像
- [ ] **テキストは意味のある長さ** - 1行ではなく、実際のユースケースを想定した長さ
- [ ] **関連テーブルも全て埋める** - メニュー、レビュー、コメント、いいね等

### レストランデータの場合

```
必須項目（全て埋める）:
- name, address, lat, lng, phone
- website_url, instagram_url, takeout_url
- tags（3つ以上）
- overview（100文字以上）
- features（全ての対応項目を記載）
- classified_images（food/interior/exterior各2枚以上）
- image_url
- is_owner_verified

関連データ:
- menus: 5品以上、全て画像・説明・価格・アレルゲン情報付き
- reviews: 5件以上、評価バラつきあり、詳細なコメント
- review_comments: 各レビューに1件以上、オーナー返信含む
- restaurant_compatibility: 全7大アレルゲンについて記載
```

---

## 2. コード作成時

### 必須チェックリスト

- [ ] **エラーハンドリングを必ず入れる**
- [ ] **型チェック・null チェックを入れる**
- [ ] **エッジケースを考慮する**
- [ ] **コメントを適切に入れる**

---

## 3. SQL/マイグレーション作成時

### 必須チェックリスト

- [ ] **実行前にDBスキーマを確認する**
- [ ] **既存のカラム・テーブルと整合性を確認する**
- [ ] **RLSポリシーを設定する**
- [ ] **インデックスを設定する**
- [ ] **一発で動作するSQLを提供する** - 分割実行は禁止

---

## 4. 報告時

### 禁止事項

- 「完了しました」→ 実際には不完全
- 「成功しました」→ 確認していない
- 「問題ありません」→ テストしていない

### 必須事項

- 実際に動作確認してから報告する
- 不明点があれば先に確認する
- 制約・エラーが起きそうな箇所は事前に対処する

---

## 違反した場合

このルールに違反した場合、ユーザーからの信頼を失う。
**常に「これで本当に完全か？」と自問してから提出すること。**
